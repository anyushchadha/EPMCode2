Field Labels

Field labels (when shown in a form) will be right aligned to the input field (unless specifically called out differently) and will never use a colon. Field labels are not bold, but are normal weight text. For field labels that need to wrap, use .fieldLabelWrap class and insert a line break within the field label to get the result shown below.

Field labels (when shown as part of a header on a data grid) will be left aligned and shown on top of the field (unless specifically called out differently) and will never use a colon. These labels are also smaller font size.

Modal Windows

Modal Windows should always have the blue title bar with a Title on the left and the Close ("X") button on the right. The title should be an <H3> tag.

A positive button (Save, Add, Go, etc..) should always be shown as a Primary Button, and should be positioned at the lower right of the dialog.

A negative button (cancel, close, exit, etc..) should always be shown as a Secondary Button, and should always be positioned at the lower left of the dialog.

Form Elements


Form Behavior

Default Focus

When a form loads, default focus is in the upper left field.


Tab Order

Tab order goes from section to section, left to right and top to bottom. So, for example, in the following screenshot, the proper tab order would be: Name (title, first, last), Nickname, Address 1, etc, all the way down the column and then on to the right column. If a section spans across columns, the tab order should be left to right and then top to bottom. Follow the sections.


Keyboard Tabbing of Radio Buttons and Checkboxes

Radio Buttons: Tab to area containing Radio Buttons, then use left, right, or top, bottom arrows to 'tab' between individual radio buttons.
Checkboxes: Tab to area contaning Checkboxes, then use Spacebar to select checkbox(s). Use Tab key to tab between individual Checkboxes.

Text Fields

Text fields should use pre-defined widths to maintain uniformity. By default, text fields have margins added to maintain consistent spacing.



Some text fields are required. In those instances a bold, red asterisk is added to the left of the field label for the required field. The required asterisk symbol has a margin added to maintain consistent spacing. In addition, the field is had a dark blue border and a light blue background color to separate it from non-required fields. If a required text field has data pre-populated, it should not have the highlight applied.

Combo Boxes / Drop-down Lists If there is one and only one item, the list should default to that item instead of requiring the user to pick a selection. e.g. in a one doctor practice, Faith Front Desk should not have to always select the doctor to make an appointment for a patient. Combo boxes that are not required should always have a blank option at the beginning of the list so that the user can "un-select" something if necessary. When the user selects the blank option - the default text should be shown in the combo box.

Radio Buttons & Checkboxes Radio buttons and checkboxes should always be shown to the left of their respective labels.
Checkboxes are used in two types of situations:
  1. for a boolean flag, such as only showing active patients
  2. when the user may want to make multiple selections and there are 6 items or less

A radio button group can either be displayed as a vertical unit or a horizontal unit. If displayed as a vertical unit, the field label for the group should be displayed at the top of the unit. This will need to be accomplished on a case by case basis.


If the radio group is displayed as a horizontal unit, all text should live on the same line.

Checkboxes will usually be displayed as a vertical unit and usually will not have an associated field label.

Text Areas

Text areas should use pre-defined widths to maintain uniformity. By default, text areas have margins added to maintain consistent spacing.

Form Validation


Messaging

Message language needs to always precisely match field label language including capitalization patterns. So if the field is called "Phone", the validation message would be something like, "Enter a valid Phone number." If the field is labeled "Contact Name" the validation is "Enter a Contact Name." Words like "Please" don't really make the software more polite and should be avoided to help focus the content of the message.

If the validation issue is that the input is malformed, tell the user how to make it right. e.g. "Enter a 9 digit number for NPI" or "License can only be alphanumeric characters", etc.

Messaging: Validation & Required Fields

Required fields are always marked with a red asterisk. In addition, if the user has not saved any data to the field previously, and the field is blank, it should be highlighted (as defined in the Text Fields section above). If, when the page loads, the field has data pre-populated (meaning the form may have been saved before, etc.), any required fields that have data already populated should not show as highlighted (but will still always show the asterisk).

Once a blank, required field has data entered in it - the highlight on the field should be removed, leaving only the red asterisk.

If the value in the field changed (blank -> data, data -> different data, data -> blank) validation should be kicked off when the user exits the field (tabs out, clicks out, etc.) If validation fails, the field should be highlighted in the error style (as defined in the Text Fields section above) and should show the appropriate error messages directly below the field. Once the errors have been corrected in the field, the error style will be removed when the user exits the field (tabs out or clicks somewhere else on the page).

If validation passes, the field should return to it's default state - with no highlighting.
